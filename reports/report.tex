\section{Abstract}
%% ============================================================

This report describes a video-based bowling score detection pipeline that accepts a mobile-recorded bowling video (typically 25--30 seconds, MP4 or MOV format) and produces a fully annotated output video. The system marks each pin from the moment it falls onward with a green circle, overlays the wall-clock timestamp (MM:SS) at which each pin fell, displays a per-frame HUD showing elapsed time and cumulative pin count, and renders a two-second summary overlay at the end of the clip. The core approach fine-tunes a \textbf{YOLOv8s} object detector on a custom bowling dataset, couples it with \textbf{ByteTrack} for stable pin identity across frames, and applies a per-pin \textbf{three-state machine} to determine fall events robustly. On the held-out validation set the detector achieves mAP@50 = 98.19\% and mAP@50-95 = 93.83\%; on the separately held-out test set it achieves mAP@50 = 97.19\% and mAP@50-95 = 91.21\%.

%% ============================================================
\section{Introduction}
%% ============================================================

Automatically determining which bowling pins have fallen, and when, is a localised instance of event detection in video. The task is harder than it first appears: the camera is handheld and non-stationary, pins share a similar appearance whether standing or fallen, multiple pins can fall simultaneously at the ball-impact frame, and recording takes place under variable indoor lighting with motion blur during the throw.

The specification for this project requires:
\begin{itemize}
  \item Detect falling pins in a recorded video and mark each one continuously from its fall time with a visible indicator.
  \item Record the wall-clock timestamp of each fall.
  \item Produce an annotated output video and a final score summary.
\end{itemize}

The approach treats the problem as object detection + multi-object tracking + event classification. Object detection (YOLOv8s) locates each pin in every frame. Multi-object tracking (ByteTrack) gives each pin a consistent integer identity across the video so its state can be monitored over time. A per-ID state machine classifies each pin as standing, potentially falling, or definitively fallen based on geometric changes to its bounding box over a 5-frame window, gated by the detected proximity of the ball. All drawing is done with OpenCV, and the output is written to a new video file at the original resolution and frame rate.

%% ============================================================
\section{System Architecture}
%% ============================================================

The pipeline has five sequential stages. Each stage receives its input from the previous one; all persistent state (pin identities, fall timestamps, HUD counters) is accumulated in a dictionary that advances with each frame.

\subsection{Stage 1 --- Video Ingestion}

OpenCV \texttt{VideoCapture} opens the input file (MP4 or MOV). Frame dimensions, frames-per-second, and total frame count are read from the container metadata. Frames are read sequentially in a loop; no preprocessing is applied before detection because YOLOv8 handles its own input normalisation internally.

\subsection{Stage 2 --- Detection}

The fine-tuned YOLOv8s model runs on each frame via \texttt{model.track()}, which combines detection and tracking in a single call. Two classes are detected: \textbf{ball} (class index 0) and \textbf{pin} (class index 1). Both standing and fallen pins are labelled under the single class \texttt{pin}; the rationale is in Section~\ref{sec:single-class}. The confidence threshold is set to 0.10 (justified in Section~\ref{sec:conf-sweep}). Detected boxes are separated into ball boxes and pin boxes for downstream processing.

\subsection{Stage 3 --- Tracking}

ByteTrack (integrated into Ultralytics as the default tracker) assigns a stable integer track ID to each pin detection. The ID persists across frames even when a pin is briefly occluded or detection confidence temporarily drops. The tracker output per frame is a set of (box, track\_id, class) tuples.

\subsection{Stage 4 --- Fall Determination}

Each track ID maintains an independent three-state machine: \textsc{Standing} $\to$ \textsc{Maybe\_Falling} $\to$ \textsc{Fallen}. Transitions are governed by geometric criteria on the bounding box (aspect ratio and centroid-drop thresholds), subject to a ball-proximity gate. Full details are in Section~\ref{sec:fall}.

\subsection{Stage 5 --- Rendering}

For each frame, OpenCV draws:
\begin{itemize}
  \item A green filled circle at the centroid of every \textsc{Fallen} pin, persistent for all subsequent frames.
  \item A MM:SS text label above each fallen-pin marker, showing the second the pin fell.
  \item A corner HUD at the top-left: current \texttt{Time: MM:SS} and \texttt{Pins down: N}.
  \item At video end: a two-second dark overlay with \texttt{GAME OVER}, total elapsed time, and total pins knocked down.
\end{itemize}
\texttt{cv2.VideoWriter} encodes the annotated frames to the output path at the original frame rate and resolution.

\section{Dataset Construction}
%% ============================================================

\subsection{Recording Protocol}
\label{sec:recording}

All videos were recorded on an iPhone at approximately 30 fps (MOV container, H.264 codec). Fifteen clips were recorded across multiple sessions in the same indoor environment. Pin layouts varied across clips: standard triangular arrangement, scattered pins for spare practice, and reduced-count line setups. Lighting was fixed indoor fluorescent. No controlled studio conditions were used.

\subsection{Frame Extraction}

Each clip was sampled with a stride of 1:20 (one frame retained per 20 frames, approximately 1.5 fps effective). This yields a manageable annotation workload while preserving both pre-impact and post-impact states in the keyframe set. The long edge of each extracted frame was resized to 1280 pixels using \texttt{INTER\_AREA} downsampling to preserve edge sharpness. In total \textbf{557 keyframes} were extracted.

\subsection{Annotation}

Frames were uploaded to Roboflow for bounding-box annotation. Annotation policy:
\begin{itemize}
  \item All pins visible in a frame --- whether standing or fallen --- were labelled as class \texttt{pin}.
  \item The ball was labelled as class \texttt{ball} in every frame where it was visible.
  \item No distinction was made between standing and fallen pin instances (see Section~\ref{sec:single-class} for reasoning).
\end{itemize}

\subsubsection*{Why custom data?}

Publicly available bowling datasets depict regulation alleys with long lanes, overhead ceiling cameras, and uniform pin spacing. The visual domain is entirely different from a handheld side-on mobile recording. A detector pre-trained or fine-tuned on that data would face a large domain gap.

\section{Critical Data Remediation}
%% ============================================================

\label{sec:remediation}

Roboflow's default export for this project produced \emph{polygon} labels (a list of $(x_i, y_i)$ vertices per annotation) rather than the axis-aligned bounding-box format that YOLOv8 requires (\texttt{class x\_center y\_center width height}, all normalised to $[0,1]$). Training on the unmodified export caused catastrophic convergence failure: loss did not decrease and the model produced no detections.

Three remediation operations were applied to all 3,051 polygon annotations:

\paragraph{Operation 1: Polygon to bounding-box conversion.}
For each polygon with vertices $\{(x_i, y_i)\}_{i=1}^{n}$, the axis-aligned bounding box is derived as:
\begin{align}
  x_{\min} &= \min_i x_i, \quad x_{\max} = \max_i x_i \notag \\
  y_{\min} &= \min_i y_i, \quad y_{\max} = \max_i y_i \notag \\
  x_c &= \tfrac{x_{\min}+x_{\max}}{2}, \quad
  y_c  = \tfrac{y_{\min}+y_{\max}}{2}, \quad
  w    = x_{\max}-x_{\min}, \quad
  h    = y_{\max}-y_{\min} \notag
\end{align}

\paragraph{Operation 2: Coordinate clamping.}
All coordinates are clamped to $[0, 1]$ to remove out-of-bounds values from polygon vertices that extended slightly beyond the image border.

\paragraph{Operation 3: Extreme-box filtering.}
Annotations with $w > 0.45$ or $h > 0.45$ (likely mislabelled whole-frame regions) or with $w < 0.005$ or $h < 0.005$ (degenerate single-pixel boxes) were removed.

\subsection{Why a Single ``Pin'' Class}
\label{sec:single-class}

An intuitive design would use two classes: \texttt{standing\_pin} and \texttt{fallen\_pin}, so that classifier output directly indicates fall state. This design was rejected for the following reason.

ByteTrack maintains a stable track ID by matching detections across consecutive frames based on IoU and appearance. When a pin begins to fall, its bounding box changes substantially in shape (tall and narrow $\to$ wide and short). If the class label also changes at that moment, the tracker treats the old detection as a disappearing object and the new detection as a new appearing object, assigning a fresh ID. This ID switch happens at precisely the frame where the fall event needs to be recorded under the original ID. The state machine would never observe the transition on a consistent identity.

Keeping a single \texttt{pin} class allows the tracker to match bounding boxes through the shape change using IoU, preserving the ID. Fall detection is then performed on the geometric properties of the box under a stable ID.

\subsection{Training Procedure}

Training ran on a Kaggle T4 GPU (16 GB VRAM). The cosine learning rate schedule reduces the learning rate smoothly toward zero, avoiding abrupt gradient steps and consistently improving final mAP in YOLO fine-tuning compared to step-LR schedules. Early stopping with \texttt{patience=20} terminates training if validation mAP does not improve for 20 consecutive epochs.

%% ============================================================
\section{Tracking}
%% ============================================================

\label{sec:tracking}

Tracking is required because the fall-detection state machine must observe a single pin's bounding box over multiple consecutive frames. Without tracking, there is no guarantee that two detections in adjacent frames refer to the same physical pin.

\textbf{ByteTrack} was selected because:
\begin{itemize}
  \item It is integrated into Ultralytics and requires no additional code: passing \texttt{tracker="bytetrack.yaml"} to \texttt{model.track()} is sufficient.
  \item It maintains identities through brief detection failures (occlusion during ball impact, motion blur) by buffering low-confidence detections rather than immediately terminating a track.
  \item It uses IoU-based association, which suits pins because they are spatially separated and do not cross paths.
\end{itemize}

A custom nearest-neighbour matcher was considered and rejected: when multiple pins fall simultaneously or the ball passes between adjacent pins, nearest-neighbour matching produces systematic ID swaps.

%% ============================================================
\section{Fall Detection Logic}
%% ============================================================

\label{sec:fall}

\subsection{Why Single-Frame Detection Fails}

Three factors make single-frame fall detection unreliable:
\begin{enumerate}
  \item \textbf{Camera perspective.} Pins falling toward or away from the camera change aspect ratio less than pins falling sideways; a single threshold may miss them entirely.
  \item \textbf{Motion blur.} The ball-impact frame typically has 2--3 frames of motion blur that degrades the detected bounding box shape.
  \item \textbf{Occlusion.} The ball or a neighbouring pin may partially occlude a falling pin, distorting the detected box dimensions during the critical frames.
\end{enumerate}

\subsection{Three-State Machine}
\label{sec:statemachine}

Each tracked pin ID maintains one of three states: \textsc{Standing}, \textsc{Maybe\_Falling}, \textsc{Fallen}. Transitions are:
\[
\textsc{Standing}
\xrightarrow{\text{criterion met (any frame)}}
\textsc{Maybe\_Falling}
\xrightarrow{\text{criterion met for 5 consecutive frames}}
\textsc{Fallen}
\]
If the criterion is \emph{not} met in \textsc{Maybe\_Falling} state, the counter resets and the pin returns to \textsc{Standing}.

The \textbf{fall criterion} for a given frame is:
\[
\frac{h}{w} < 0.9 \quad \textbf{OR} \quad \Delta y_c > 0.5 \times h_0
\]
where $h/w$ is the current bounding-box aspect ratio, $\Delta y_c$ is the downward displacement of the centroid from its standing reference position, and $h_0$ is the box height in the \textsc{Standing} reference frame. Either condition alone is sufficient.

\subsection{Ball-Proximity Gate}
\label{sec:gate}

A pin can enter \textsc{Maybe\_Falling} only if the ball was detected within $2 \times h_{\mathrm{pin}}$ of the pin centroid in the previous 60 frames. This prevents spurious fall detections from camera shake, a person walking past, or annotation artefacts. If the ball detector fails for more than 60 consecutive frames, the gate is lifted.

\section{Known Limitations}
%% ============================================================

\label{sec:limitations}

\begin{enumerate}
  \item \textbf{Axial falls.} Pins falling toward or away from the camera change aspect ratio minimally. The centroid-drop criterion compensates, but it is a weaker signal and may require more frames before the 5-frame sustain threshold is reached.

  \item \textbf{Tracker ID switches during ball impact.} Heavy occlusion in the one or two frames around ball contact can cause ByteTrack to drop a pin's track and assign a new ID on reappearance. If the new ID starts in \textsc{Standing} state, the fall event under the original ID is logged correctly but the persistent marker switches to the new ID.

  \item \textbf{Motion blur on fall frames.} The 2--3 frames of motion blur immediately around pin contact degrade bounding box shape accuracy. The 5-frame sustain threshold mitigates this at the cost of a small detection lag ($\leq$5 frames).

  \item \textbf{Ball-proximity gate bypass.} If the ball detector fails continuously for more than 60 frames (e.g.\ the ball leaves the frame or is fully occluded), the ball gate is lifted and any subsequent anomalous pin movement can trigger a false fall.

  \item \textbf{Single-environment dataset.} All 15 recording clips were captured in the same room under similar lighting conditions. Performance on significantly different backgrounds, lighting, camera angles, or alley types has not been tested.
\end{enumerate}

%% ============================================================
\section{Conclusion}
%% ============================================================

\label{sec:conclusion}

This project delivers a five-stage pipeline that processes a mobile-recorded bowling video and produces a fully annotated output video with per-pin fall markers, MM:SS timestamps, a per-frame HUD, and a final score summary. The fine-tuned YOLOv8s detector meets the accuracy target, achieving mAP@50 = 98.19\% on the validation set and 97.19\% on the separately held-out test set. The combination of ByteTrack for stable pin identity and a 3-state fall machine with a 5-frame sustain threshold and ball-proximity gate produces reliable fall event detection under the recording conditions in the dataset.

The system runs post-hoc on a recorded clip, not in real time. The five failure modes in Section~\ref{sec:limitations} represent genuine open problems, particularly for novel recording environments outside the training distribution. Extending the dataset to include multiple locations, camera heights, and lighting conditions is the most direct route to improving robustness.
